# Project Improvements

## Цель файла
Собрать практичные предложения по улучшению проекта DJgurda Bot, чтобы использовать их как технический backlog.

## Приоритет P0 (критично)

### 1. Безопасность секретов и локальных файлов
- Проблема: локальные секреты и служебные данные могут попадать в рабочий контекст.
- Действие:
  - Проверить `.gitignore` и исключить все чувствительные локальные артефакты.
  - Убедиться, что `.env` не содержит посторонних строк и хранит только env-переменные.
  - Ввести ротацию токенов при любом подозрении на утечку.
- Результат: снижение риска компрометации доступа к боту и инфраструктуре.

### 2. Валидация env перед запуском деплоя
- Проблема: ошибки конфигурации обнаруживаются уже в runtime контейнера.
- Действие:
  - Добавить preflight-проверку в `manager.sh` (проверка обязательных переменных в `ENV_FILE`).
  - Явно логировать отсутствующие ключи перед `docker run`.
- Результат: более предсказуемый деплой и меньше аварийных запусков.

## Приоритет P1 (высокий)

### 3. Разграничение прав на команды управления
- Проблема: `/start` и `/stop` доступны всем участникам чата (сейчас это осознанно).
- Действие:
  - Определить политику доступа: оставить как есть или ограничить администраторами.
  - Если ограничивать: добавить проверку роли через Telegram API.
- Результат: контроль над поведением бота в активных чатах.

### 4. Стратегия по неактивным handlers
- Проблема: в коде есть YouTube/Instagram/YandexMusic/VK handlers, но в runtime подключен только TikTok.
- Действие:
  - Выбрать подход: поэтапное включение или архивирование неиспользуемых обработчиков.
  - Для включаемых источников добавить критерии readiness и чеклист.
- Результат: меньше технического долга и понятный roadmap по источникам.

### 5. Улучшение наблюдаемости деплоя
- Проблема: есть базовые логи, но нет структурной диагностики по шагам.
- Действие:
  - Добавить в `manager.sh` явные этапы и код завершения каждого шага.
  - Добавить краткую сводку успешного деплоя (image tag, env, время запуска).
- Результат: быстрее поиск проблем после релизов.

## Приоритет P2 (средний)

### 6. Улучшение конфигурационного контракта
- Проблема: `src/config.py` требует токены даже для неактивных интеграций.
- Действие:
  - Разделить обязательные и условно-обязательные env-переменные.
  - Для неактивных интеграций добавить lazy-проверки только при их использовании.
- Результат: проще локальный запуск и меньше ложных падений на старте.

### 7. Формализация release notes
- Проблема: изменения деплоя и структуры иногда фиксируются только в коммитах.
- Действие:
  - Ввести короткий шаблон release notes на каждую заметную правку инфраструктуры.
  - Добавить ссылку на него из `README.md`/`.github/ai_context.md`.
- Результат: быстрее онбординг и прозрачность изменений.

## Приоритет P3 (дальше)

### 8. Возврат к тестам как отдельная фаза
- Проблема: `tests/` сейчас legacy и не участвует в актуальном контуре.
- Действие:
  - При готовности вернуть минимальный smoke-набор:
    - startup/shutdown,
    - middleware gate,
    - базовая обработка URL.
  - Запускать эти тесты в CI для `dev`.
- Результат: снижение регрессий при развитии функционала.
